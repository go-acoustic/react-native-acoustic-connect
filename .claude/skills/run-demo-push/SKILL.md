# Run Demo (Push) — Acoustic Connect samples (iOS + Android)

Guides running the sample apps and debugging the push and build issues that come
up. Two samples share one SDK and one push-troubleshooting playbook:

- **`Examples/bare-workflow/`** — bare React Native (npm + CocoaPods + Gradle).
- **`Examples/expo/`** — Expo (managed prebuild / EAS) via the bundled Config
  Plugin (`plugin/`, `app.plugin.js`).

The diagnostic playbook and the platform troubleshooting references below apply
to **both** — the SDK and push behaviour are identical. Only the run flow and a
few Expo-specific gotchas differ (see [Run Flow](#run-flow)). Each sample has its
own `README.md` "Troubleshooting" section, and both lean on the shared SDK setup
CLI (`acoustic-connect doctor`).

## When to Use

Use this skill when the user wants to **run the bare-workflow or the Expo demo**
(`expo prebuild` / `expo run:ios|android`, EAS build issues) or is hitting any of
these (iOS or Android, either sample):

- "no push token" / "session reaches the collector but no push messages"
- "app crashes on launch" (`EXC_BREAKPOINT`)
- "no rich image" / "NCE doesn't render" / long-press shows nothing
- "thumbnail not showing"
- `processDebugGoogleServices … No matching client` (Android FCM)
- `PushService is not initialized` / `connect-push-fcm on classpath: false`
- "token registers but no notification arrives"
- `pod install` / `fmt` / Xcode / SDK-version build errors
- "how do I run this on a device vs simulator/emulator"

## Run Flow

### bare-workflow (`Examples/bare-workflow/`)

```bash
cd Examples/bare-workflow

# 1. One-command setup (doctor + full auto): prerequisites, config scaffolding
#    from templates, validation, installs. Idempotent. Delegates config checks
#    to the shared SDK CLI (acoustic-connect doctor) and the iOS push-extension
#    wiring to `acoustic-connect setup-ios-push`.
npm run bootstrap            # auto-detect; or bootstrap:ios / bootstrap:android

# 2. Metro
npm run start

# 3. Build + launch
npm run ios                  # macOS
npm run android
```

Bootstrap copies any missing per-developer config from the committed templates
(`ConnectConfig.json` ← `ConnectConfig.example.json`; iOS `Signing.local.xcconfig`;
Android `google-services.json`) and tells you exactly what to fill in. It never
contains real credentials — the sample is published to partners.

### Expo (`Examples/expo/`)

```bash
cd Examples/expo

# 1. Doctor + scaffold config (same shared SDK CLI). Idempotent.
npm run bootstrap            # → acoustic-connect doctor + npm install

# 2. Generate native projects from the Config Plugin, then build + launch
npx expo prebuild            # writes ios/ + android/ (NSE/NCE, entitlements, Podfile, gradle)
npx expo run:ios --device    # or: npx expo run:android
```

Differences from bare-workflow that change how you debug it:

- **NSE/NCE are generated by the Config Plugin on `prebuild`** — *not* by
  `acoustic-connect setup-ios-push`. To repair the push extensions, re-run
  `expo prebuild` (or fix `plugin/`), not the bare scaffolder.
- The App Group / entitlements / Podfile / Android `config.gradle` apply are all
  written at prebuild from `ConnectConfig.json` + the plugin props — don't
  hand-edit the generated `ios/`/`android/` (they're gitignored CNG output).
- `expo run:ios`'s install step can crash in Expo's `LockdowndClient` even though
  the build succeeded. Workaround: `xcrun devicectl device install app --device
  <UDID> <built .app>`, then `npx expo start --dev-client`.
- **EAS:** `.easignore` excludes `Examples/expo/ios|android` so EAS does a managed
  prebuild instead of bare-building stale local native dirs. Don't commit them.

Expo-specific push gotchas (APNs environment vs code signing, fresh-install token
churn, NSE log capture) are documented in `Examples/expo/README.md`
"Troubleshooting". Everything in the Diagnostic Playbook and the platform
references below applies unchanged.

## Device vs Simulator / Emulator (read before debugging push)

- **iOS Simulator** (Xcode 14+/Apple Silicon) **does** register for APNs, receive
  remote pushes, and run the NSE/NCE. The **one** thing it does not do reliably is
  render the **collapsed-banner rich-media thumbnail** — validate that specific
  visual on a **physical device**. (Do not repeat the "simulator can't get a push
  token" myth — it can.)
- **Android emulator** must be a **Google Play** system image (has
  `com.google.android.gms` + `com.android.vending`) to receive FCM. A plain AOSP
  AVD will never get a token.

## Diagnostic Playbook

The first question for any push problem: **is registration failing, or is
delivery failing?** They look identical from the app ("I sent a push, nothing
happened") but have completely different fixes.

- **Registration** = the device obtains a token and the SDK posts a
  `pushRegistration` (**message type 22**, with `mobileToken` and
  `mobileProvider` `APN`/`FCM`) to the Collector. Verify this in the logs first.
- **Delivery** = the backend sends through APNs/FCM and the device receives it.
  If registration works but no notification arrives, the problem is backend
  targeting / credentials, **not** the app.

Platform specifics — read the matching reference file:

- iOS → [troubleshooting.ios.md](troubleshooting.ios.md)
  (signing/no-token, NSE/NCE log capture, crash-report triage, framework-link
  checks, entitlements/`.xcent`).
- Android → [troubleshooting.android.md](troubleshooting.android.md)
  (logcat filters, `connect-push-fcm` classpath, `PushService` init,
  google-services package match, Firebase "Send test message" isolation,
  POST_NOTIFICATIONS, Play-services emulator).

## Validate a Fix (runtime, not just build)

Always confirm both build **and** runtime. For push specifically:

1. Build + launch; confirm no crash (`~/Library/Logs/DiagnosticReports` on macOS,
   `adb logcat` crash trace on Android).
2. Capture the SDK push logs (see the platform reference) and confirm a
   **type-22 `pushRegistration` with a real `mobileToken`** reaches the Collector.
3. Send a real push and confirm receipt. Rich-media (thumbnail + expanded image)
   requires a **physical device** on iOS.

## Known Fixed Defects (document, don't re-fix)

These were root-caused and fixed previously — if they recur, the named place is
where to look:

- iOS CLI no-token → `DEVELOPMENT_TEAM` via `ios/Signing.local.xcconfig`.
- iOS NCE crash on expand → NCE must link `UserNotificationsUI.framework`.
  bare-workflow: wired by `acoustic-connect setup-ios-push` /
  `cli/ios/add_push_extensions.rb`. Expo: the Config Plugin's NCE
  target must link it (`plugin/src/withConnectNCE.ts`) — re-run `expo prebuild`
  to repair.
- Android FCM not registering → Firebase BoM + `firebase-messaging` in
  `android/build.gradle`, and `Connect.push.enable(...)` at init in
  `HybridAcousticConnectRN.kt`.
